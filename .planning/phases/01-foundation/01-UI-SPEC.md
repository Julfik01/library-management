# Phase 1 — UI Design Contract

> Visual and interaction contract for the Foundation phase (AUTH-01–AUTH-07).
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

---

## Scope

Phase 1 delivers **four screens** and one shared shell:

| Screen | Who Sees It | Requirement |
|--------|------------|-------------|
| Register | Unauthenticated users (students only self-register) | AUTH-01 |
| Login | All unauthenticated users | AUTH-02 |
| Dashboard shell | Authenticated users — role-routes on load | AUTH-03, AUTH-07 |
| Create Librarian form | Admin librarian only | AUTH-06 |
| Unauthorized (403) | Any user hitting a forbidden route | AUTH-07 |

The Dashboard shell is a layout container with a top nav and a role-appropriate welcome state. No data tables or catalogs — those belong to Phase 2+.

## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, input icon padding, badge internal gap |
| sm | 8px | Inline form field gap, button icon-to-label gap |
| md | 16px | Default element spacing, card internal padding |
| lg | 24px | Form section padding, card padding on desktop |
| xl | 32px | Auth card vertical padding |
| 2xl | 48px | Page-level vertical breathing room above auth card |
| 3xl | 64px | Centering offset — top padding on auth pages |

Exceptions:
- Touch targets (buttons, links): minimum 44px height. shadcn `Button` default is `h-10` (40px); use `h-11` (44px) on mobile breakpoint for auth form submit buttons.
- Auth card max-width: 400px (not a spacing token — a layout constraint).

## Color

Phase 1 uses **Neutral** as the shadcn base color (selected during `shadcn init`). All values below are the CSS variable names that shadcn sets in `globals.css`. Executor must not hardcode hex values — reference CSS variables only.

| Role | shadcn CSS Variable | Tailwind Class | Usage |
|------|--------------------|--------------:|-------|
| Dominant (60%) | `--background` | `bg-background` | Page background, auth page fill |
| Secondary (30%) | `--card` / `--muted` | `bg-card`, `bg-muted` | Auth card surface, nav bar background, form field background |
| Accent (10%) | `--primary` | `bg-primary`, `text-primary` | See reserved list below |
| Destructive | `--destructive` | `bg-destructive`, `text-destructive` | Error messages on form fields only in Phase 1 |

**Accent (`--primary`) reserved exclusively for:**
1. The primary submit button on each form (Register, Login, Create Librarian)
2. Active nav item indicator (left border or underline)

Accent must NOT be applied to: labels, body text, card borders, helper text, secondary actions.

**Light/dark mode:** Do not implement dark mode in Phase 1. shadcn ships both light and dark CSS variables — leave them in place but do not add a theme toggle. Dark mode is deferred to post-v1.

## Screen Contracts

### Screen 1: Login (`/login`)

**Layout:** Centered auth card, white surface on gray background, vertically centered on viewport.

**Structure:**
```
Page background (bg-background)
  └── Auth Card (max-w-[400px], bg-card, rounded-lg, shadow-md, p-xl)
        ├── Heading: "Sign in" (20px, semibold, text-foreground)
        ├── Subheading: "Enter your email and password to access your account." (14px, regular, text-muted-foreground)
        ├── [Form]
        │     ├── Email field: Label "Email" + Input type="email"
        │     ├── Password field: Label "Password" + Input type="password"
        │     ├── [Alert — visible only on error]
        │     └── Submit Button: "Sign in" (full-width, variant="default", h-11)
        └── Footer link: "Don't have an account? Register" (14px, text-primary, text-center)
```

**States:**
- Default: empty form, no alert visible
- Loading: Submit button shows spinner + disabled; label changes to "Signing in..."
- Error (401): Alert variant="destructive", copy: "Invalid email or password. Please try again."
- Error (network): Alert variant="destructive", copy: "Unable to connect. Check your internet connection and try again."

**Navigation after success:**
- role = 'student' → `/dashboard`
- role = 'librarian' → `/dashboard`
- role = 'admin_librarian' → `/dashboard`
All roles land on the same `/dashboard` route; the dashboard shell renders role-appropriate content.

**Layout:** Identical centered auth card pattern as Login.

**Structure:**
```
Auth Card
  ├── Heading: "Create account" (20px, semibold)
  ├── Subheading: "Register to access the university library." (14px, muted)
  ├── [Form]
  │     ├── Full Name field: Label "Full name" + Input type="text"
  │     ├── Email field: Label "Email" + Input type="email"
  │     ├── Password field: Label "Password" + Input type="password" + helper: "Minimum 8 characters"
  │     ├── [Alert — visible only on error]
  │     └── Submit Button: "Create account" (full-width, variant="default", h-11)
  └── Footer link: "Already have an account? Sign in" (14px, text-primary, text-center)
```

**States:**
- Default: empty form
- Field validation (inline, on blur): red border + helper text below field (shadcn Form uses `FormMessage`)
- Loading: Button disabled + spinner + "Creating account..."
- Error (409 conflict): Alert variant="destructive", copy: "An account with this email already exists. Sign in instead."
- Success: Redirect to `/dashboard`

**Validation rules (Zod schema — executor implements):**
- full_name: required, min 2 chars, max 100 chars
- email: required, valid email format
- password: required, min 8 chars

## Interaction Contracts

### Form Validation Timing

- **On submit:** Run full Zod schema validation. Show all field errors simultaneously.
- **On blur (after first submit attempt):** Revalidate individual field. Show/clear per-field error.
- **On change (before first submit):** No validation — do not show errors while user is typing for the first time.

Implementation: `react-hook-form` with `mode: "onTouched"` and `reValidateMode: "onChange"`.

### Session Initialization (App Bootstrap)

On every app load (`App.tsx`):
1. Show full-page Skeleton.
2. POST /auth/refresh (httpOnly cookie sent automatically by browser).
3. Success → store access_token in AuthContext → render authenticated layout.
4. 401 response → clear AuthContext → redirect to /login.
5. Network error → redirect to /login (do not loop).

This is the AUTH-03 requirement. Executor must implement this in `App.tsx` using `useEffect` + TanStack Query's `useQuery` or a bare `axios` call before QueryClient is ready.

### Token Refresh on 401

Axios response interceptor (implemented in `frontend/src/lib/axios.ts`, Pattern 5 from RESEARCH.md):
- On 401 from any protected API call → attempt POST /auth/refresh → retry original request.
- On refresh failure → clear AuthContext → redirect to /login.
- Queue concurrent 401s during refresh (RESEARCH.md Pattern 5 handles this).

### Protected Route Behavior

`ProtectedRoute` component checks `accessToken` from AuthContext:
- No token → `<Navigate to="/login" replace />` (never flash protected content)
- Token present, wrong role → `<Navigate to="/unauthorized" replace />`
- Token present, correct role → render `<Outlet />`

### Logout

User clicks "Sign out" in nav dropdown:
1. POST /auth/logout (no loading state needed; optimistic).
2. Call `clearAuth()` from AuthContext immediately (do not wait for API response).
3. Navigate to /login.
4. If API call fails, session still clears on client side (best-effort server invalidation).

## Accessibility Contract

- All form inputs use `<label>` via shadcn `<Label>` with `htmlFor` matching input `id`.
- Error messages associated with inputs via `aria-describedby` (shadcn `<FormMessage>` handles this automatically when using `<FormField>`).
- Submit buttons use native `<button type="submit">` (shadcn `Button` default).
- Focus order: natural DOM order. Do not use `tabindex` > 0.
- Loading states: submit button uses `aria-disabled` + `aria-busy` when loading. Do not use HTML `disabled` attribute alone (screen readers skip disabled elements).
- Color is never the sole indicator of an error — always pair red border with icon or text message.
