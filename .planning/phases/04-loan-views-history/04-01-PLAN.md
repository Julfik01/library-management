## Phase Goal

**As a** signed-in student or librarian, **I want to** view my loans and search loan records from one Loans page, **so that** I can track due dates and review history without leaving the app.

<objective>
Build the backend read surface that powers the Loans page: student active/history reads, librarian search, and pagination metadata.

Purpose: Phase 4 depends on backend query correctness before the UI can render the loan ledger.
Output: `backend/app/schemas/loan.py`, `backend/app/services/loan_service.py`, `backend/app/routers/loans.py`, `backend/app/main.py`, `backend/tests/test_loan_service.py`, and `backend/tests/test_loans.py`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API | Untrusted role and query input crosses into the loan read endpoints. |
| API → DB | Search and pagination queries must not leak other users' loans or reorder rows unpredictably. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing / Tampering | `backend/app/routers/loans.py` | mitigate | Enforce `require_role("librarian", "admin_librarian")` on search and current-user scoping on student reads. |
| T-04-02 | Information Disclosure | `backend/app/services/loan_service.py` | mitigate | Build joins and filters in SQLAlchemy before pagination; never slice in the client or return unscoped rows. |
| T-04-03 | Denial of Service | `backend/app/services/loan_service.py` | mitigate | Bound `page_size`, compute `total_pages`, and use stable `order_by` before `limit/offset`. |
| T-04-SC | Supply Chain | package installs | accept | No new packages are introduced in this phase; reuse the audited Phase 1 stack. |
</threat_model>
